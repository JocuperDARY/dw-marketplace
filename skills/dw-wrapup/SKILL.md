# 收尾与知识持久化

> 本 skill 是 [development-workflow](../development-workflow/SKILL.md) 的子模块，覆盖 **阶段七：收尾与知识持久化**。遵循铁律 A3（Agent 不自执行 commit）。

---

## 文档更新清单

| 文档 | 更新内容 |
|------|---------|
| 项目概要文档 | 新增条目：根因摘要、修复位置、量化前后对比、受影响文件列表 |
| API 使用指南 | 新增"已知坑位"章节：每个坑位包含错误现象、根因、正确用法代码模板 |
| 变更日志 | 记录每次重大修复的日期、根因、修复、测试结果 |
| 优化报告 | 每次性能优化编写独立报告：文件命名 `OPT_{YYYY-MM-DD}_{brief_description}.md`，含优化动机（profiling 数据支撑）、优化方法、代码变更（文件:行号）、功能等价验证、性能加速比数据、后续建议 |

---

## 知识持久化

在可访问的持久化知识库（`memory` MCP）中存储：

| 知识类型 | 内容 | 示例 |
|---------|------|------|
| **Bug 记录** | 根因机制、修复方案、如何防止回归 | "缓冲区类型不匹配导致全零输出" |
| **经验教训** | 从修复过程中发现的通用模式 | "自检需含恒零检测" |
| **工作规则** | 从具体经验中提炼的通用约束 | "新字段接入必须同步添加 FAIL 级哨兵" |

---

## 提交准备

- 所有变更文件一次性 staging
- 提交信息使用约定式格式：`<类型>: <简要描述>`
- 正文列举每个根因及其严重度
- 提交前让操作者审批（铁律 A3）

---

## 工作历史同步

将改动内容记录到项目工作历史文档（如 `SYNOPSIS.md`），确保后续查阅能理解：
- 做了什么改动
- 为什么做这个改动
- 结果如何（量化指标）
- 涉及哪些文件

---

## 基线对比记录模板

每次改动的量化验证结果必须按以下格式记录在改动对应的文档或 commit message 中：

```markdown
## 基线对比记录

| 指标 | 改动前 | 改动后 | 变化 | 判定 |
|------|--------|--------|------|------|
| 响应时间 P50  | 120ms            | 45ms             | -62.5% | ✅ 通过 |
| 响应时间 P99  | 850ms            | 210ms            | -75.3% | ✅ 通过 |
| 吞吐量        | 800 req/s        | 2400 req/s       | +200%  | ✅ 通过 |
| 错误率        | 0.5%             | 0.5%             | 持平   | ✅ 通过 |
| 峰值内存      | 512 MB           | 548 MB           | +7%    | ⚠️ 接受 |

**对比条件**：
- 测试数据：生产流量回放（5000 请求，含正常/异常/边界用例）
- 配置参数：Cache-Aside 模式，TTL=300s
- 随机种子：42
- 硬件：4 vCPU, 8 GB RAM, SSD
- 评估脚本：`scripts/benchmark.sh`

**基线 commit**：`a1b2c3d`（改动前最后一次 commit 的 hash）
```

> **原则**：缺少基线对比的验证是不完整的验证。至少记录 1-2 个关键指标。

## 收尾阶段反模式

| 反模式 | 后果 | 正确做法 |
|--------|------|---------|
| 只修代码不写文档 | 几个月后没人知道为什么这样写 | 更新项目概要和 API 使用指南 |
| 知识留在对话里 | 下次会话从零开始 | 写入 `memory` 持久化知识库 |
| 自动提交代码 | 操作者不知道改了什么 | 提交前呈交操作者审批 |
| 不记录基线对比 | 无法证明改动有效 | 至少记录 1-2 个关键指标的改前改后数据 |
