# 开发工作准则与操作流程

**版本**: 2.0 | **适用范围**: 通用（不限于本项目）

> 本 skill 融合了 `development-workflow` v1.4 与 `AI 辅助软件开发方法论`，覆盖从诊断→方案→实现→数据完整性→测试闸门→验证→收尾的完整开发循环。

---

## 概述

在系统优化与功能开发中，每一次代码改动都是对既有系统的扰动。本 skill 的目标是在「快速迭代」和「系统稳定性」之间建立硬约束——用可量化的验证替代直觉判断，用结构化的流程降低回溯成本。

**核心信念**：
- 「看起来对了」是 bug 的温床 —— **没有量化的验证等于没有验证**。
- 「先动手再说」是返工的起点 —— **想清楚再动手，写清楚再编码**。
- 「肯定是环境/方案/库的问题」是排查的天敌 —— **任何异常输出，首先假设是自己的实现错了**。这个假设在 90% 的情况下是正确的。

本方法论提炼自真实项目的完整开发-调试-验证-维护循环，面向 AI 编码助手协作场景，泛化至任何语言/框架/项目类型。

## 铁律体系

铁律分为两类：**工具使用铁律**（编辑纪律与安全）和 **流程铁律**（过程纪律）。

### 一、工具使用铁律

```
铁律 A1：所有代码修改必须使用 Edit 工具。
         —— 绕过 Edit 的修改不可追溯，会导致项目损坏无法回溯。
         例外：新建文件用 Write，删除文件用 Bash rm。

铁律 A2：保护敏感隐私信息，不得以任何形式泄露。
         —— MCP 调用前检查参数是否含凭据；不打印完整 Key；
         不上传项目数据到未授权第三方。

铁律 A3：Agent 不得自行执行 git commit。
         —— 所有 commit 由用户亲自审核后执行。Agent 只能在用户明确授意下
         准备 commit 内容（staged files + message），不得绕过用户直接提交。
         例外：用户口头授权的一次性 commit（如 "commit this"）不在此限。

铁律 A4：最大化利用工具，不得仅凭 Read/Edit/Grep/Bash 完成所有工作。
         —— Claude Code 安装了丰富的 plugin、MCP server 和 skill，它们是
         工作流的强制组成部分，不是可选装饰。

         执行细则：
         a) 先盘点再动手：每个任务开始前，确认当前环境有哪些 plugin/MCP/skill
            可用、各自用途是什么。不得凭记忆猜测工具是否存在。
         b) 匹配即调用：工具能力与当前步骤匹配时，必须调用。不允许"这次简单，
            我手动做更快"的合理化跳过——跳过的代价是后续更高的纠错成本。
         c) 规范流程、预防风险：工具帮你 enforce 标准（如 code-review 查逻辑
            缺陷、security-review 查漏洞）。跳过工具 = 让漏洞进入下游 = 返工
            成本数十倍于调用成本。
         d) 多工具重叠时的决策：
            · 常规任务：选择最适配的一个
            · 关键任务（安全/架构/跨模块）：启动多个工具并行审查，综合决策
            · 不确定时：宁可多用，不可漏用
```

### 二、流程铁律

```
铁律 B1：计划先行，执行在后。
         —— 任何非平凡代码变更必须先写操作指引（至少含背景、方案、步骤、验收标准），
         获得共识后再动手。没有指引的实现是盲目的实现。

铁律 B2：TODO 与计划严格对照。
         —— 每一条计划子步骤必须在 TODO 列表中有对应条目，不可缺漏。
         验证时每一步行动必须能追溯到计划的某个具体子步骤。

铁律 B3：发现偏差立即分级修复。
         —— 每完成一步即与计划对照；偏差按严重度（P0/P1/P2/P3）分级。
         修复流程：先读计划→理解→修复方法论→实施→再读计划→验证。

铁律 B4：所有代码变更必须过测试闸门。
         —— 写测试→写实现→跑测试→风格检查→代码审查→安全扫描，全绿才进下一阶段。
         单次闸门不通过→修复；连续 3 次不通过→暂停，重新评估方案可行性。

铁律 B5：保护数据完整性。
         —— 哨兵值与真实值严格区分（0 是合法测量值，不是"未测量"）。
         每个分析/绘图脚本必须含 FAIL 级自检（关键字段全零→硬阻断）。
```

## 阶段门控模型

每一阶段有明确的**准入条件**和**准出条件**。前一阶段未通过准出，绝不进入下一阶段。

```
诊断 ─────→ 方案 ─────→ 实现 ─────→ 数据完整性 ─────→ 测试闸门 ─────→ 验证 ─────→ 收尾
  │           │           │              │                │              │            │
  ├─准出:     ├─准出:     ├─准出:        ├─准出:          ├─准出:        ├─准出:      ├─准出:
  │ 根因确认  │ 计划获批  │ 测试全绿     │ 哨兵就位       │ 6 闸全绿      │ 偏差清零    │ 知识落盘
  │ 证据充分  │ 回退就绪  │ 覆盖率≥80%   │ 自检通过       │              │ 三C全过     │ 提交就绪
  │           │           │              │                │              │            │
  ▼           ▼           ▼              ▼                ▼              ▼            ▼
复现+根因   操作指引     TDD循环       哨兵+元数据      分层闸门      偏差修复循环   文档+memory
            先行                         +自检                          +三C判定
```

**门控规则**：
- 诊断阶段未确认根因 → 不得进入方案设计（症状修复是浪费时间）
- 方案未经评审 → 不得进入实现（方向性错误代价远大于评审时间）
- 实现未通过测试闸门 → 不得进入验证
- 验证发现偏差 → 回到实现阶段修复，不得进入收尾
- 连续 3 个循环未通过同一闸门 → **暂停，重新评估方案可行性**

## 何时使用

对**任何**代码改动都应当遵循本流程：
- 新功能/新模块添加
- Bug 修复
- 性能优化
- 代码重构
- 参数调整
- 配置变更

**尤其**在以下情况更要严格执行：
- 改动涉及多个文件（≥3 个）
- 改动涉及核心数据流（业务核心模块）
- 改动的效果需要事后评估验证
- 时间紧、压力大（紧急时更容易出错）

**可以简化**的情况（但仍需执行核心步骤）：
- 单行/单文件的明显修复（如 typo、常量值修正）
- 纯注释/文档修改
- 仅修改配置默认值

## 阶段一：诊断与根因定位

> **核心原则：在找到根因之前，绝不实施任何修复。** 症状修复是浪费时间——它掩盖真正的问题，引入新 bug，且不防止问题回归。

### 诊断流程

#### 第 1 步：收集多维度证据

同时从以下角度平行收集数据，不依赖单一信息源：

| 角度 | 方法 | 工具 |
|------|------|------|
| 数据输出 | 检查输出文件中的异常值（全零、全 NaN、恒定值） | `Bash` — 直接分析 CSV/JSON |
| 可视化产物 | 查看生成的图片/视频，检查视觉异常 | `Read`（图片）/ `playwright` MCP |
| 日志 | 搜索 ERROR/WARNING/异常栈 | `Grep` — 搜索日志文件 |
| 配置 | 对比不同运行组的配置差异 | `Bash` — diff 工具 |
| 数据流 | 从输出反查输入，逐层追踪上游数据源 | `codegraph_explore` MCP |
| API 文档 | 查证第三方库/API 的输入输出格式和约定 | `context7` MCP（resolve + query-docs） |

#### 第 2 步：建立假设并交叉验证

- 对每个候选根因写出明确假设："我认为 X 是根因，因为 Y 证据"
- 用多个独立来源的证据交叉验证同一假设
- 区分"相关"与"因果"——多个根因可能叠加

#### 第 3 步：区分根因层级

将确认的根因分类：

| 层级 | 定义 | 示例 |
|------|------|------|
| **致命** | 导致核心功能完全失效 | 缓冲区类型不匹配导致全零输出 |
| **严重** | 导致功能部分失效或结果错误 | 算法方向反转导致结果符号全错 |
| **数据陈旧** | 历史数据由旧版本代码生成，不可修复 | 诊断字段在新代码实现前全为空 |
| **呈现缺陷** | 数据正确但展示误导 | 不适用的指标画成平直线而非标注不可用 |

#### 第 4 步：评估修复优先级

按影响范围和可修复性排序。优先修数据生成侧 bug（修复后所有下游受益），其次修数据收集侧 bug，最后修呈现侧 bug。

### 诊断反模式

- ❌ 看到"全零"就加 `if zero: fix()`——先找出为什么是零
- ❌ 一个症状对应一个 fix——多个症状可能共享同一根因
- ❌ 只检查代码不检查数据——输出文件是最客观的证据
- ❌ 修复第一个找到的 bug 就停——叠加 bug 是常态

### 操作指引示例（填好的）

以下是一个通用的后端优化场景示例：

```markdown
# 操作指引：为 API 添加 Redis 缓存层

## 1. 背景与动机

### 1.1 问题描述
用户列表页接口 `/api/users` 每次请求都查询数据库，平均响应时间 850ms。
在峰值 200 QPS 时，数据库连接池被打满，P99 延迟上升到 3200ms。
业务特征：用户数据变更频率低（约 5% 的请求会触发写操作）。

### 1.2 问题特征分析
这是一个**读多写少**的典型场景：读请求占 95%、写仅占 5%。
核心矛盾是数据库 IO 成为瓶颈，而缓存可以通过空间换时间大幅降低读延迟。
本质特征：数据一致性要求不苛刻（可以接受数秒的延迟窗口）。

## 2. 方案探索

### 方案 A：本地内存缓存（LRU）
- **核心思路**：在应用进程内维护一个 LRU 缓存，定时过期
- **优点**：无外部依赖、零网络开销、实现简单（Go: `lru` / Python: `@lru_cache`）
- **缺点**：多实例部署时缓存不一致、进程重启后缓存丢失、内存占用随列表数量线性增长
- **取舍点**：简单快速 vs 多实例一致性

### 方案 B：分布式 Redis 缓存
- **核心思路**：引入 Redis 作为独立缓存层，所有实例共享同一缓存
- **优点**：多实例一致性、TTL 自动过期、持久化可配、支持集群
- **缺点**：引入新中间件（运维成本）、增加一次网络跳转（~1ms）、需防护缓存穿透/击穿/雪崩
- **取舍点**：一致性与扩展性更好 vs 架构复杂度增加

### 方案 C：CDN + 边缘缓存
- **核心思路**：通过 CDN 缓存完整的 API 响应
- **缺点**：用户维度数据无法做个性化缓存；缓存粒度太粗（整个响应而非数据）；
  不适合动态查询参数场景
- **取舍点**：适用于静态/公共内容，不适用于动态 API

## 3. 选定方案与理由

**选择**：方案 B（Redis），采用 Cache-Aside 模式

**理由**：
- 需要支持多实例部署，方案 A 的多实例不一致无法接受
- 读多写少 + 数据一致性要求不苛刻 → Cache-Aside 足够
- 已有运维基础设施支持 Redis

**舍弃的方案**：
- 方案 A：多实例部署后缓存不一致会导致数据展示错误
- 方案 C：API 含分页参数(`?page=2&size=20`)，CDN 无法处理动态查询

## 4. 实现计划

- [ ] 步骤 1：抽象缓存接口 `Cache[K, V]`，包含 `Get/Set/Delete` 三个方法
  - 验收标准：接口定义完整，编译通过
  - 涉及文件：`pkg/cache/interface.go`
  - 预估改动：1 个接口定义，~20 行

- [ ] 步骤 2：实现 Redis 适配器
  - 验收标准：单测覆盖正常 Set/Get/过期/连接断开 四种场景
  - 涉及文件：`pkg/cache/redis.go`, `pkg/cache/redis_test.go`
  - 预估改动：~150 行实现 + ~80 行单测

- [ ] 步骤 3：在 UserService 中集成 Cache-Aside
  - 验收标准：读请求先查缓存→miss 后查 DB→回写缓存；写请求清除缓存
  - 涉及文件：`internal/service/user.go`
  - 预估改动：~30 行

- [ ] 步骤 4：压力测试验证
  - 验收标准：响应时间 P50 < 50ms, P99 < 200ms, 缓存命中率 > 85%
  - 涉及文件：无（运行压测脚本）

## 5. 回退计划

### 失败判定条件
- 缓存命中率 < 50%（说明缓存的数据集不匹配访问模式）
- 引入缓存后 P99 延迟反而上升（说明 Redis 网络开销 > DB 查询收益）

### 回退操作
- 通过配置开关 `CACHE_ENABLED=false` 一键降级为直连 DB
- 降级后观察 24 小时，确认无误后再清理代码

### 保留的中间产出
- 压测报告（基线 vs 缓存 vs 降级 三组数据）
- 接口文档中缓存语义的说明
```

> 以上是一个完整填写的示例。在实际使用中，直接复制上方模板，参考本例填充。

| 阶段 | 占实现周期 | 产出物 |
|------|-----------|--------|
| 发散（方案探索） | 15-25% | 候选方案 + 优缺点 |
| 收敛（选定 + 计划） | 10-15% | 操作指引文档 |
| 实现 | 40-50% | 可运行代码 |
| 验证 | 20-30% | 量化验证报告 |

### 计划粒度标准

- 每个子步骤应可在 10-30 分钟内完成
- 每个子步骤必须有可验证的产出物（测试通过、文件存在、输出值在范围内）
- 分层分阶段：先核心修复 → 再基础设施 → 最后呈现优化

### 回退策略

- 每阶段完成前备份可回退的中间产物
- 冒烟先行：投入全量资源前，先用最小规模实验（1 个数据单元、1 个配置）验证
- 明确"不可回退"的步骤（如覆盖生产数据），提前确认

## 阶段三：TDD 实现

### RED-GREEN-REFACTOR 循环

```
  写失败测试(RED)
       │
       ▼
  确认测试因预期原因失败（非语法/导入错误）
       │
       ▼
  写最小实现(GREEN)
       │
       ▼
  确认所有测试通过
       │
       ▼
  重构(REFACTOR) —— 代码清晰化
       │
       ▼
  确认测试仍通过
```

### 测试分层策略

| 层级 | 范围 | 执行时间 | 覆盖目标 |
|------|------|---------|---------|
| **纯函数单元** | 单个无副作用的函数 | <1ms | 逻辑正确性、边界条件、格式转换 |
| **集成单元** | 带 mock 的类方法 | <100ms | API 调用正确性、状态转换 |
| **硬件相关** | 需真实硬件的测试 | 1-10s | 端到端数据流验证 |
| **冒烟** | 完整管线的子集 | 分钟级 | 文件 I/O、格式正确性 |

### 测试设计要求

- **纯函数测试第一优先**：能用纯函数验证的逻辑绝不依赖特定硬件/数据库/网络
- **合成数据优于真实数据**：已知 ground truth 的合成序列比真实数据更可靠
- **每层测试独立可运行**：硬件相关测试用标记隔离，无对应环境自动跳过
- **测试应验证"不应该发生的事"**：如关键输出字段不应为全零、关键诊断字段不应全为哨兵值——这些是防止静默退化的终极哨兵

### 实现纪律

1. 严格按照操作指引中的 TODO list 逐项实现
2. **所有代码修改必须通过 Edit 工具**（铁律 A1）
3. 保护敏感信息（铁律 A2）
4. 实现过程中发现计划遗漏 → 先更新指引再继续

## 阶段四：数据完整性防护

### 哨兵值策略

- 定义明确的"不适用/未测量"标记值（如 `-1`），与"测量值为零"区分
- 数据收集端：未计算的指标写入哨兵值，**绝不写入 0**（0 是合法的测量值）
- 数据消费端：聚合前过滤哨兵值，每个图表面板检查"是否所有数据都是哨兵值"

### 元数据落盘

**运行时元数据**：每次运行在输出目录写入元数据文件，包含：

| 字段 | 用途 |
|------|------|
| 后端/引擎标识 | 下游根据后端类型做条件渲染 |
| 配置参数关键值 | 复现运行条件 |
| 代码版本哈希 | 追溯生成数据的代码版本 |
| 运行时间戳 | 区分多次运行 |

**用途**：下游分析脚本根据元数据做条件渲染（如"某类后端不显示特定指标"），不再靠猜测或启发式推断。

### 自检哨兵

在每个分析/绘图脚本的输出阶段添加自检。分两级：

**FAIL 级（硬阻断）—— 必须包含的检查**：

| 检查项 | 含义 |
|--------|------|
| 关键字段是否全程为 0 | 如流场幅度全零 → 后端故障 |
| 关键字段是否全程为哨兵值 | 如诊断字段全为 -1 → 旧代码产物 |
| 数据行数是否为零 | 如所有输入文件为空 → 数据收集失败 |

**WARN 级（软提示）—— 建议包含的检查**：

| 检查项 | 含义 |
|--------|------|
| 子分类合计与总数不一致 | 数据聚合逻辑可能有误 |
| 均值/中位数比过大 | 偏态严重，需关注 |
| 某类字段全部可用或全部不可用 | 可能是过滤条件过于激进 |

## 阶段五：分层测试闸门

在代码变更完成后，按顺序通过以下闸门才能进入验证阶段：

### 闸门检查清单

| 序号 | 闸门 | 通过标准 |
|------|------|---------|
| 1 | 单元测试 | 全部通过 |
| 2 | 硬件/集成测试 | 全部通过或全部因环境缺失而跳过 |
| 3 | 风格检查 | 无新增违规 |
| 4 | 类型检查 | 无新增错误 |
| 5 | 代码审查 | 无致命/高危问题 |
| 6 | 安全扫描 | 无致命问题 |

### 闸门失败处理

| 级别 | 含义 | 行动 |
|------|------|------|
| **致命** | 安全漏洞、数据丢失风险 | **阻断**——必须修复后才能继续 |
| **高危** | Bug 或显著质量问题 | **警告**——应在当前阶段修复 |
| **中危** | 可维护性问题 | **记录**——可在下一阶段修复或记录为技术债 |
| **低危** | 风格或次要建议 | **备注**——可选修复 |

> 连续 3 次修复后闸门仍不通过 → **暂停，重新评估方案可行性**（铁律 B4）。

## 阶段六：系统性验证与偏差修复

### 验证循环（计划→实现逐项对照）

实施完成后，执行完整的计划→实现对照验证：

```
FOR EACH 计划子步骤:
    1. 定位计划中的具体描述
    2. 用 codegraph_explore 定位对应的实现代码块（精确到文件:行号）
    3. 验证: 功能是否完全对应？逻辑是否一致？
    4. 标注: 完全匹配 / 部分匹配（注明差异） / 缺失
    5. 汇总偏差清单，按严重度分级（见 §偏差严重度分级）

FOR EACH 偏差:
    1. 重读计划中的对应段落，用自己的话复述理解
    2. 写出修复方法论：改哪些文件、怎么改、预期结果
    3. 按方法论实施修复
    4. 重读计划对应段落再次对照
    5. 运行相关测试确认
```

### 三 C 判定标准

#### 验证金字塔（三层，从底到顶）

```
        ┌──────────┐
        │  集成验证  │  ← 端到端运行，确认整体指标未退化
       ┌┴──────────┴┐
      ┌┴────────────┴┐
      │  功能验证     │  ← 断言 / 中间态输出，确认逻辑分支正确
     ┌┴──────────────┴┐
     │  编译/语法验证   │  ← import 通过、无语法错误
     └────────────────┘
```

- **编译/语法验证**：`python -c "import module"`、类型检查。**不得跳过。**
- **功能验证**：断言测试、新旧结果对比、中间态打印校验
- **集成验证**：运行完整 pipeline，确认产出指标在预期范围内

#### 三 C 判定

| C | 名称 | 检查项 | 验证方法 |
|---|------|--------|---------|
| 1 | **Consistency（一致性）** | 实现与操作指引一致？无「惊喜」逻辑？参数/配置与设计一致？ | 逐条对照 TODO list |
| 2 | **Completeness（完整性）** | TODO 全部实现？边界/异常/错误处理齐全？配套产出齐全？回退/关闭机制存在？ | Checklist 逐项打勾 |
| 3 | **Correctness（正确性）** | 编译/导入通过？功能/集成验证通过？量化指标在预期范围？ | 量化对比 + 基线对比 |

**三项必须同时通过，缺一不可。**

#### 三 C 循环

```
        开始验证
            │
            ▼
      ┌──────────────┐
      │  一致性检查    │ ← 不通过 → 回到实现修正偏离
      └──────┬───────┘
             ▼ 通过
      ┌──────────────┐
      │  完整性检查    │ ← 不通过 → 补充遗漏实现
      └──────┬───────┘
             ▼ 通过
      ┌──────────────┐
      │  正确性检查    │ ← 不通过 → 修复（先查实现错误）
      └──────┬───────┘
             ▼ 全部通过
          验证通过 ✅
```

- 单次未通过 → 回到阶段三修改
- 指引本身有问题（逻辑矛盾/物理不可行）→ 回到阶段二修订指引
- **连续循环 3 次未通过 → 暂停，重新评估方案可行性**

### 偏差严重度分级

| 级别 | 定义 | 行动 |
|------|------|------|
| **P0** | 阻塞性偏差——核心功能缺失或错误 | 立即修复，暂停其他工作 |
| **P1** | 重要偏差——影响验证完整性或数据质量 | 当前阶段修复 |
| **P2** | 次要偏差——文档、注释、测试覆盖不足 | 下阶段修复或记录为技术债 |
| **P3** | 外部阻塞——需人工操作或特定硬件/网络依赖 | 明确标注并提供操作指引 |

### 功能等价验证层次（性能优化改动专属）

性能优化在三 C 之外还需验证**功能等价**——优化前后输入相同则输出必须相同：

| 级别 | 测试内容 | 方法 | 接受标准 |
|------|---------|------|---------|
| L1 | 单元等价 | 对优化函数用相同输入调用新旧版本 | 输出完全一致（浮点 `atol=1e-5`） |
| L2 | 集成等价 | 在完整管线上对比优化前后的输出产物 | 输出文件逐值一致 |
| L3 | 回归等价 | 运行项目评估流程对比报告 | 核心指标数值一致 |

**浮点差异处理**：向量化与 JIT 因运算顺序不同可能引入 ≤1e-6 的差异，属正常范围。超过此值需追查。

### 实测驱动调试

当单元测试通过但集成/硬件测试失败时，进入**实测驱动调试**模式：

1. **最小复现脚本**：编写独立脚本（<50 行），绕过所有项目框架代码，直接调用底层 API
2. **对照实验设计**：每次只改变一个变量，其他条件完全相同
3. **观察→假设→验证循环**：每一轮实验产出一个明确结论
4. **证据链完整**：从实验结论→代码修改→测试通过，每一步都可追溯

### 测试阈值校准

- 合成数据的 ground truth 与算法输出之间存在固有误差（算法不可能完美恢复精确变换）
- 测试阈值应设为：`误差 < 容差系数 × 期望幅度`
- 实测后根据实际误差调整阈值，**但阈值必须保留**——防止回归到全零状态

### 验证命令示例（Python 项目）

以下命令可以直接套用到大多数项目中：

```bash
# ── 编译/语法验证 ──

# 验证模块能否正常导入
python -c "import mymodule; import mylib"

# 编译检查（不执行）
python -m py_compile mymodule.py

# 类型检查（如项目使用类型注解）
# mypy mymodule.py


# ── 功能验证（断言测试） ──

# 验证关键函数的输入输出
python -c "
from mylib import process_data
import numpy as np

# 构造最小输入
input_data = np.array([[1.0, 2.0], [3.0, 4.0]])

# 调用被测函数
result = process_data(input_data, option='default')

# 断言
assert result is not None, 'Expected non-None result'
assert len(result) == 2, f'Expected 2 items, got {len(result)}'
print(f'Result: {result}')
"

# 验证边界条件（空输入、极值）
python -c "
from mylib import process_data
import numpy as np

# 空数组
result = process_data(np.empty((0, 2)), option='default')
assert result is not None, 'Should handle empty input gracefully'
print('Empty input test passed')

# 单元素
result = process_data(np.array([[5.0, 5.0]]), option='default')
assert result is not None, 'Should handle single element'
print('Single element test passed')
"


# ── 集成验证（端到端） ──

# 对核心入口函数做空运行
python -c "
from myapp import pipeline

# 使用最小测试数据测试 pipeline 能否正常跑完
pipeline(
    input_path='test_data/small_sample',
    output_dir='test_output',
    dry_run=True,  # 空运行模式
)
print('Pipeline end-to-end OK')
"
```

> **提示**：修改代码后，至少运行上述三类验证的各一个命令。全部通过后再进入下一阶段。

## 核心方法

### 量化验证方法目录

#### 性能类

| 方法 | 适用场景 | 工具 |
|------|---------|------|
| FPS 对比 | 速度优化 | `time`、`cProfile`、自定义计时 |
| 内存 profiling | 内存优化 | `memory_profiler`、`tracemalloc` |
| CPU/GPU 利用率 | 资源利用 | `nvidia-smi`、`py-spy` |
| 延迟分位数 | 实时系统 | P50/P95/P99 统计 |

#### 优化核心原则

任何优化必须遵循以下五条原则（从项目优化实践中提炼）：

1. **功能等价（Functional Equivalence）**：优化不得改变原有行为。相同输入必须产生相同输出。
2. **可替换性（Drop-in Replacement）**：优化后的代码应能无缝替换原代码，接口不变。
3. **可验证性（Verifiability）**：每次优化必须附带验证手段——功能测试确认行为不变，性能测试确认加速真实。
4. **渐进优化（Incremental）**：优先优化热点路径。不做无目标的提前优化。
5. **可读性折中（Readability Trade-off）**：优化不应过度损害可读性。显著牺牲可读性的优化需附加注释说明意图。

#### 优化决策树

在决定是否优化以及如何优化前，按以下顺序逐级评估：

```
1. 是否有性能问题？               → 无 → 不优化（保持现状）
   ↓ 是
2. 是否是热点路径（≥总耗时 10%）？ → 否 → 标记暂缓，不投入时间
   ↓ 是
3. 是否有更优的算法？             → 是 → 算法优化（复杂度降级/预计算/早期退出）
   ↓ 否
4. 是否有更优的库可替代？         → 是 → 库替换优化（向量化/更快的数据格式）
   ↓ 否
5. 是否适合 JIT 编译？           → 是 → JIT 优化（热路径循环 10-100x 加速）
   ↓ 否
6. 是否有不必要的 IO？           → 是 → IO 优化（批量读写/延迟加载/条件跳过）
   ↓ 否
7. 是否可以并发/并行？           → 是 → 并发并行优化（CPU密集→多进程, IO密集→线程）
   ↓ 否
8. 是否可以 GPU 加速？          → 是 → GPU 优化（合并内核启动/减少传输）
   ↓ 否
9. 保持现状，标记未来关注
```

**原则**：无 profiling 不优化。先用性能分析工具定位热点（≥总耗时 10% 才值得优化），再按树逐级评估。不可跳过 profiling 直接优化非热点代码。跳过 profiling 的优化通常优化了占比 <5% 的代码，净收益微乎其微。

#### 精度类

| 方法 | 适用场景 | 说明 |
|------|---------|------|
| 数值对比（MSE/MAE） | 回归/滤波 | 与旧版逐帧数值差异 |
| 指标对比（F1/准确率/吞吐量等） | 项目核心指标 | 同口径同数据同条件对比 |
| A/B 测试 | 参数调优 | 一次改一个参数 |
| 消融实验 | 组件评估 | 逐个开关组件量化贡献 |

#### 信号级诊断

| 方法 | 适用场景 | 说明 |
|------|---------|------|
| 中间态信号检查 | 复合系统 | 检查管道是否产生预期中间量（而非只看最终指标）如 enabled 标志、中间变量值 |
| 逐组件隔离验证 | 管道型系统 | 每个子组件独立测试其输入输出，而不是一次跑整条管道 |
| 诊断可视化 | 模式识别 | 将关键信号绘制成图表，识别肉眼难察的模式：分布、突变、周期性 |

#### 跨数据集交叉验证

| 维度 | 验证内容 | 操作 |
|------|---------|------|
| 场景覆盖 | 不同特征的输入数据 | 至少覆盖 2-3 个特征差异较大的场景 |
| 预期行为验证 | 验证各场景上的行为是否符合预期 | 场景 A 不触发、场景 B 触发、场景 C 部分触发 |
| 退化检测 | 改动了场景 A 的逻辑但只在场景 B 上测 | 所有受影响的场景都要回测 |

### 中间态信号检查——细说

实践中最容易犯的错误是只盯着最终指标（如响应时间、准确率），忽略了中间管道是否产生了正确的信号。

| 阶段 | 最终指标 | 中间态信号 | 本应发现的问题 |
|------|---------|-----------|-------------|
| 初版实现 | 无提升 | `feature_enabled=0/377(0%)` | 新功能从未激活——死代码 |
| 第 2 版 | 无提升 | `feature_enabled=0`, `input_ready=0` | 参数未传入核心函数 |
| 第 3 版 | 无提升 | `feature_enabled=120/377(32%)`, `input_ready=205` | 管道活了！但为什么结果全零？|
| 第 4 版 | 无提升 | 中间量全零且无错误日志 | 中间量从未被赋值——诊断代码缺失 |

最终指标不动，就认为"方案没用"然后放弃——但中间信号根本没产生过。

#### 信号检查清单

每次改复合系统时，不仅要检查最终指标，还要按顺序检查：

```
□ 管道是否被调用？（增加 print/日志确认入口执行）
□ 输入是否正确？（检查输入数据内容，而非 shape/type）
□ 中间量是否合理？（范围、分布、趋势）
□ 输出是否被正确写回？（下游是否读取了预期字段）
□ 最终指标是否变化？
```

> **原则**：中间态信号是管道内第一响应者。最终指标是滞后的。优先检查信号，而非指标。

### 反向验证法：诊断先行

最有效的验证方式不是"看代码对不对"，而是**设置诊断变量 → 运行 → 查看诊断输出 → 反向推导组件是否正常运行**。

#### 为什么反向验证比代码审查更可靠

```
代码审查模式：
  代码是对的 → 代码应该产生 X 结果 → 运行验证 → 结果不对 → 重新审查代码
  （假设前提：你能通过看代码判断它是否正确）

反向验证模式：
  设置诊断变量 → 运行 → 看产出 →
    ├─ 结果显示预期信号 → 管道正常 ✅
    └─ 结果显示异常信号 → 追查具体层级
  （不依赖对代码正确性的预判——让数据说话）
```

典型例子：一段复杂的数据处理管道，代码审查认为"函数实现了，应该在工作"。但设置诊断变量后运行发现中间变量全为零——原因是参数传递链断裂，核心函数从未被实际调用。这些问题在代码审查中不会暴露，因为它们都是运行时的间接问题。

#### 如何实施反向验证

```
□ 步骤 1：确定关键诊断信号
   └─ 什么中间量能直接反映组件是否在工作？
   └─ 例：某功能 → enabled 率 / 输入量 / 输出量 / 质量指标

□ 步骤 2：确保诊断代码完整
   └─ 每个代码路径都要写入诊断变量
   └─ 常见遗漏：提前返回/异常退出时没写诊断
   └─ 建议在函数开始时初始化所有诊断字段为默认值

□ 步骤 3：构建可视化
   └─ 将诊断信号绘制成图（分布直方图、时序折线图）
   └─ 可视化能暴露数值审查中忽略的模式

□ 步骤 4：建立预期基准
   └─ 在不同场景下定义"正常"的中间量范围
   └─ 例：静态场景某参数≈0, 运动场景该参数非零

□ 步骤 5：迭代验证
   └─ 改一次代码 → 跑诊断 → 看输出 → 确认行为符合预期
   └─ 不符合 → 追查 → 修复 → 重复步骤 5
   └─ 符合 → 运行最终评估确认指标
```

#### 诊断变量的设计原则

| 原则 | 说明 | 反例 |
|------|------|------|
| **每个路径都记录** | 函数的所有返回路径都必须设置诊断 | 提前 `return` 不写诊断变量 → 显示 0 |
| **用值域可解释的量** | 用有物理意义的量（百分比、计数、比率） | 用复杂的内部状态叠起来无法解读 |
| **时间序列优先** | 按时序记录，可以画折线图观察趋势 | 只输出均值会掩盖中间波动 |
| **区分"未激活"和"激活后为零"** | 用两个不同信号表示 | `enabled=0` vs `enabled=1, output=0` |
| **极端情况下也有输出** | 即使出错也记录错误原因 | 静默崩溃 → 下游不知道发生了什么 |

### 闭环验证：检测指标 → 负反馈 → 修正

验证不是一次性的——它应是一个闭环：**设计检测指标 → 设置捕获点 → 运行 → 检查信号 → 如果异常（负反馈）→ 从链路起点重新排查 → 修复 → 再运行**。

```
         ┌──────────────────────────────────┐
         │  设计检测指标                      │
         │  └─ 哪些中间量能反映组件是否工作？   │
         └──────────┬───────────────────────┘
                    ▼
         ┌──────────────────────────────────┐
         │  设置捕获点                        │  ← 每级变换留下验证痕迹
         │  └─ 每级输入输出都记录              │
         └──────────┬───────────────────────┘
                    ▼
         ┌──────────────────────────────────┐
         │  运行                             │
         │  └─ 执行管道产生数据                │
         └──────────┬───────────────────────┘
                    ▼
         ┌──────────────────────────────────┐
         │  检查信号                          │  ← 对比预期基准
         │  ├─ 信号正常？→ 继续到下一阶段       │
         │  └─ 信号异常？→ 负反馈触发          │
         └──────────┬───────────────────────┘
                    ▼
         ┌──────────────────────────────────┐
         │  负反馈：从链路起点逐级排查           │  ← 绝不从结论倒退
         │  └─ 级 1 正常？→ 级 2              │
         │  └─ 级 1 异常？→ 修复级 1          │
         │     └─ 只相信本级诊断数据           │
         └──────────┬───────────────────────┘
                    ▼
         ┌──────────────────────────────────┐
         │  修复 → 再运行 → 确认信号正常       │
         └──────────────────────────────────┘
```

#### 两种反馈回路

| 回路 | 触发条件 | 响应 | 目的 |
|------|---------|------|------|
| **正反馈**（确认） | 信号在预期范围内 | 继续到下一阶段或任务结束 | 保证进度向前 |
| **负反馈**（修正） | 信号偏离预期 | 回到链路起点重新排查 | 防止 bug 累积 |

**负反馈回路是质量的核心。** 它要求：
- 不跳过任何一级排查
- 不因"看起来问题不大"而放过异常信号
- 不因"已经排查了 3 级"而省略第 4 级
- 发现异常 → 必须追溯到根因 → 修复 → 再运行确认

#### 闭环排查示例

当中间信号异常时，**绝不允许跳级排查**。一次真实案例中，一个异常信号（输出全零）被错误归因为"方案本身不适用"（P3），跳过了藏在链路中的 4 个实现 bug（P1）。

正确流程：
```
输出全零 → 负反馈触发
  └─ 级 1：输入数据是否存在？→ 是（200+ 条）
  └─ 级 2：数据过滤掩码是否正确？→ 不对！新数据未过滤
      └─ bug 1：补数方法没更新掩码 → 修复
  └─ 再运行 → 输出有值但偏小
  └─ 级 3：统计字段存了什么？→ 子集数，不是总数
      └─ bug 2：赋值语义错误 → 修复
  └─ 再运行 → 输出正常
  └─ 级 4：状态标记 → 只有默认值
      └─ bug 3：回退路径没写标记 → 修复
  └─ 级 5：异常路径 → 4 条遗漏诊断
      └─ bug 4-6：补充赋值
  └─ 全部修复 → 验证通过
```

### 全生命周期逐级排查法

"中间态信号检查"告诉你**查什么**，本方法告诉你**怎么逐级查**——从数据管道的起点到终点，每级验证后才进入下一级。

#### 与信号检查的区别

```
信号检查：      查找关键信号 → 确定异常区域 → 修复（目标：找到异常）
生命周期排查：  明确数据链路 → 每级捕获输入/输出 → 逐级验证
               └─ 目标不是"找到异常"，而是"每级都正常"才往下走
```

#### 通用生命周期模板

以一条典型的数据处理管道为例——数据从创建、变换、合并、到最终消费，经过多级处理。当你需要排查管道异常时，为每一级回答三个问题：

```
□ 阶段 N：该阶段的转换名称
   ├─ 输入是什么？（类型、范围、来源）
   ├─ 输出是什么？（类型、范围、目标）
   └─ 捕获点：输入量是否在预期范围？输出量是否合理？中间量是否有意义？

逐级验证顺序：
  输入创建 → 数据提取 → 合并/变换 → 质量过滤 → 核心算法 → 结果消费
      ↓          ↓          ↓          ↓          ↓          ↓
   类型正确?   提取量达标?  合并率正常?  过滤率合理?  输出非零?   下游可见?
```

#### 关键原则

1. **每级必须有捕获点**——排查阶段每级至少有一个 `print`/`log` 确认输入输出
2. **捕获点在开发阶段就设置**——不要等出了问题再想在哪捕获
3. **捕获点的产出物可以丢弃**——开发阶段的详尽量化日志可以在验证通过后移除
4. **"正常"是一个范围，不是固定值**——不要因为"和之前的值不一样"就判定异常，要理解该级数据的合理波动范围

### 常见实现错误分类

**归因优先级**：当结果异常时，按以下优先级逐级排查——绝不允许跳级：

```
优先级 1：实现错误（90% 的异常来自此类）
   ├─ 传参错误（调用链中参数被默认值覆盖）
   ├─ 索引/边界错误（数组越界、守卫条件失效）
   ├─ 条件/逻辑错误（and/or 混用、分支条件写反）
   ├─ 维度/类型错误（shape 不匹配、float32 vs float64）
   ├─ 累积/状态错误（状态未在关键帧/重置点清零）
   └─ 诊断遗漏（功能实现了但诊断字段未赋值 → 输出全零）

优先级 2：方案理解偏差
   └─ 方案依赖的假设在当前场景是否成立？参数语义是否理解正确？

优先级 3：方案本身不适用
   └─ 只有彻底排除 1 和 2 后才能考虑「方案不行」
```

**核心教训**：多次实践中，最终指标异常时直接跳到「方案本身不适用」的结论，跳过了藏在链路中的多个实现 bug。正确做法是先假设实现有错，逐级排查。

90% 的异常来自实现错误而非方案缺陷。按以下分类自查：

| 类别 | 典型表现 | 检查方法 |
|------|---------|---------|
| 传参错误 | 调用链中参数被默认值覆盖、参数名不匹配 | 逐层打印实际接收的参数值 |
| 索引/边界错误 | 数组越界、长度变化后索引失效 | 检查守卫条件 + 边界值测试 |
| 条件/逻辑错误 | `and`/`or` 混用、取反遗漏、分支条件写反 | 分支覆盖测试（每个分支一个用例） |
| 维度/类型错误 | 数组维度不匹配、`float32` vs `float64` | 打印 `.shape` + `.dtype` |
| 累积/状态错误 | 帧间状态未在关键帧/重置点清零 | 检查重置逻辑是否覆盖所有状态字段 |
| 诊断遗漏 | 功能实现了但诊断字段未赋值（输出全零） | 搜索诊断输出字段的每个写入点 |
| 路径/配置错误 | 路径拼接、配置键名拼写、环境变量未设置 | 运行前打印配置解析结果 |

### 实验登记表示例

当尝试多个方向但都没有收益时，按以下流程记录和决策：

```
□ 方向登记：每个方向记录名称、假设、预期收益
□ 最小实现：每个方向只做最小可验证实现（不完善但可测）
□ 量化判决：每个方向跑相同条件的评估，记录关键指标
□ 放弃条件：预设放弃条件（如指标下降或关键信号不合理）
□ 结论留存：即使是失败的方向，保留实验结论（避免重复尝试）
```

| # | 方向 | 假设 | 最小改动 | 实测指标 | 关键信号 | 判决 | 放弃原因 |
|---|------|------|---------|---------|---------|------|---------|
| 1 | 方案 A | 假设描述 | ~80 行 | 0.69 | enabled=32%, output=0 | ❌ 放弃 | 方案未实际生效 |
| 2 | 方案 B | 假设描述 | ~30 行 | 0.68 | 信号噪声大 | ❌ 放弃 | 输入质量不满足假设 |
| 3 | 方案 C | 假设描述 | ~40 行 | 0.69 | enabled 不变 | ❌ 放弃 | 死循环：依赖前置条件 |
| 4 | 方案 D | 假设描述 | ~50 行 | 0.65 | 关键指标下降 | ❌ 放弃 | 精度低于现有方案 |
| 5 | 方案 E | 假设描述 | ~3 行 | 0.69 | 无退化 | ✅ 保留 | 零开销，无害 |
| 6 | 方案 F | 假设描述 | ~30 行 | 0.61→默认关 | 参数过大 | ✅ 保留 | 默认关，按需启用 |

**关键教训**：每次尝试都记录判决和理由。避免在 6 个月后回来发现有人又在试相同的方案。

### 管道未激活检查

当怀疑某段代码"没有被执行"时（这是一个极其常见的问题），按以下顺序排查：

```
□ 入口条件是否满足？（if guard、配置开关、hasattr 检查）
□ 函数是否被显式调用？（搜索函数名是否存在调用点）
□ 参数是否被正确传递？
   └─ 从调用链最外层逐层检查参数传递链
   └─ 任意一层默认值覆盖都会导致下游收不到正确值
□ 返回值是否被接收或使用？（检查调用处是否解包返回值）
□ 诊断代码是否完整？（功能实现了但诊断记录没写——输出全零的经典案例）
```

> **可悲的真相**：90% 的「方案无效」其实是「方案根本没运行」。

### 版本回退策略

当发现某个提交引入了回归时：

```
□ 定位有问题的提交范围（git log --oneline，找到引入点）
□ 确定回退粒度（整个提交？还是只回退特定文件？）
  └─ 如果是大量文件的混合提交，只回退受影响的关键文件
  └─ `git show <commit> --stat` 查看改动范围和类型
□ 选择回退方式：
   方法 A: `git revert <commit>` — 保留历史，适合多人协作
   方法 B: `git checkout <commit>^ -- <file>` — 只回退特定文件（不丢失其他改动）
□ 保留有问题的代码的分析结论供后续参考
```

本 skill 不孤立运行，与其他 skill 组成完整工作链：

```
                         ┌─────────────────┐
                         │  brainstorming    │ ← 需求模糊时先发散思路
                         └────────┬────────┘
                                  ▼
                         ┌─────────────────┐
  ┌──────────────────────│ development-     │ ← 本 skill：诊断→方案→实现→验证→收尾
  │                      │ workflow         │
  │                      └────────┬────────┘
  │          ┌────────────────────┼────────────────────┐
  │          ▼                    ▼                    ▼
  │  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
  │  │ systematic-   │    │ code-review  │    │ verify        │
  │  │ debugging     │    │              │    │               │
  │  └──────────────┘    └──────────────┘    └──────────────┘
  │         ▲                   ▲                    ▲
  │         │                   │                    │
  └─────────┴───────────────────┴────────────────────┘
     三 C 验证中正确性不通过时，      一致性/完整性检查    集成验证环节
     按调试仪式排查无果 → 调用      通过后做 code-       需要看实际运行
     systematic-debugging 深入追查   review 检查逻辑缺陷  效果 → 调用 verify
```

| 触发条件 | 调用的 skill | 作用 |

## 工具普查与编排

> **前置铁律**：铁律 A4 要求每个工作阶段都必须积极调用工具。本节描述如何系统性地盘点工具、分层归类、按任务选配——这是执行铁律 A4 的操作手册。

**为什么必须普查**：

- 工具在持续新增/更新，凭记忆选型会遗漏更优解。
- 同一任务可能被多个工具覆盖（如多个 skill 都能做规划设计），需要根据任务规模和复杂度选配。
- 多智能体并行场景下，不同 agent 类型有不同专长（如通用审查 vs 语言专项审查 vs 安全审查），错配导致质量下降。

### 普查流程

```
□ 步骤 1：列出现有环境所有可用辅助工具
   └─ 包括：skill 列表、MCP server 及其 tools、可用的 agent/subagent 类型
□ 步骤 2：按能力分层归类（见下方分层框架）
□ 步骤 3：根据任务特征，从每层选配 1-3 个最匹配的工具
□ 步骤 4：设计编排顺序（串行/并行/流水线）
```

### 能力分层框架

以下框架描述的是**能力类别**，而非具体工具。在实际环境中，将可用工具按此框架归类：

#### 规划/设计层

需要寻找具备以下能力的工具：

| 能力 | 描述 | 适用场景 |
|---|---|---|
| 结构化发散 | 引导式提问→多方案探索→收敛→设计文档 | 需求模糊、方案不明 |
| 计划生成 | 将设计转为 WBS 分解、任务列表、依赖图 | 设计确认后、实现前 |
| 代码库架构分析 | 分析现有代码模式，输出文件/接口/数据流蓝图 | 多文件改动、需理解现有架构 |
| 系统架构决策 | 技术选型、可扩展性评估、取舍分析 | 跨模块架构决策 |
| 多视角评估 | 并行出多套方案 + 综合评分 | 重大改动需要多角度评估 |

#### 实现层

| 能力 | 描述 | 适用场景 |
|---|---|---|
| 测试驱动开发 | 红-绿-重构循环，强制测试先行 | 新功能、bug 修复 |
| 语言/框架惯用法 | 特定语言的最佳实践、设计模式、代码组织 | 模块/类/接口设计 |

#### 并行/多智能体层

| 能力 | 描述 | 适用场景 |
|---|---|---|
| 独立任务并行调度 | 将无状态依赖的独立子任务分派给多个 agent 并行执行 | 多模块独立实现 |
| 依赖感知分派 | 按 DAG 顺序逐个分派 agent，后置任务等待前置完成 | 有依赖顺序的多步骤实现 |

| 构建修复 | 编译/类型/依赖错误的诊断和修复 | 构建失败时 |

| 能力 | 描述 | 适用场景 |
|---|---|---|
| 完成前验证清单 | 系统化的完成前检查（三 C 等） | 改动完成的最后关卡 |

|---|---|---|

| 能力 | 描述 | 适用场景 |
|---|---|---|
| 文档生成 | 从代码自动生成 README、DESIGN 骨架 | 新模块完成后 |

### 编排模板

根据任务规模和复杂度，选择以下编排模式：

#### 轻量改动（单文件、<30 行）

```
规划（可选） → 实现 → 审查 → 运行验证
```

#### 标准改动（多文件、新功能/模块）

```
规划设计 → 计划生成 → 实现 → 审查 → 完成前验证
                       └─ TDD（若新增逻辑）
```

#### 大型改动（3+ 模块、跨层影响）

```
规划设计 → 计划生成
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
 并行实现    并行实现    并行实现
 (模块 A)    (模块 B)    (模块 C)
    │           │           │
    └───────────┼───────────┘
                ▼
        并行审查 × N
                │
                ▼
         完成前验证
```

**编排铁律**：

1. **无状态依赖 → 并行**：两个模块独立实现、不共享状态时并行启动。
2. **有依赖顺序 → 串行**：A 的输出是 B 的输入时必须等 A 完成。
3. **不确定是否有隐式依赖 → 先探后定**：用代码探索工具追踪数据流，确认无隐式耦合后再并行。
4. **审查在实现后立即进行**：不等所有模块完成。每个模块完成后即触发对应能力的审查工具。

## 场景化快速参考

不同改动类型可以裁剪流程中的环节，但以下环节**不得裁剪**：

| 改动类型 | 必须执行 | 可以简化 | 典型耗时 |
|---------|---------|---------|---------|
| **Bug 修复** | 诊断（阶段 1）+ 操作指引背景分析 + 最小复现用例 + TDD + 功能验证 + 闸门 | 集成验证可只跑子集；指引可精简 | 1-3 小时 |
| **性能优化** | 基线对比 + 量化验证（FPS/内存）+ 功能等价验证（L1-L3） | 操作指引可精简，方案探索 1-2 个即可 | 2-4 小时 |
| **新功能/模块** | 完整操作指引（全部 5 部分）+ 全部 7 阶段 | 方案探索可收缩到 2-3 个 | 1-5 天 |
| **重构（不改变行为）** | 编译验证 + 集成验证（跑分不能变）+ 三 C 一致性 | 操作指引可以很简短；不需要量化精度类验证 | 1-2 天 |
| **参数调优** | 基线对比 + A/B 测试 + 消融实验 | 不需要操作指引；直接记录对比表 | 2-8 小时 |
| **配置变更** | 编译验证 + 主路径功能验证 | 无需操作指引和完整三 C | 15-30 分钟 |
| **纯文档/注释** | 无（走常规流程即可） | — | 5-15 分钟 |

### Bug 修复 — 快速流程

```
□ 诊断：收集多维度证据 → 建立假设 → 确认根因（阶段 1）
□ 写背景分析（原因 + 现象）
□ 构造最小复现用例
□ 编写精简操作指引（阶段 2）
□ TDD 实现（先写复现测试 RED → 修复 GREEN → 重构）（阶段 3）
□ 功能验证（断言或对比输出）
□ 走过闸门（阶段 5）
□ 集成验证（子集跑分确认不引入 regression）（阶段 6）
□ 同步到工作历史 + 知识持久化（阶段 7）
```

### 性能优化 — 快速流程

```
□ 记录基线指标（FPS / 内存 / 精度）
□ 方案探索（1-2 个方案）
□ 实现（Edit 工具）
□ 量化验证（与基线同口径对比）
□ 功能等价验证（L1→L2→L3）
□ 同步到工作历史
```

### 新功能 — 完整流程

```
□ 编写完整操作指引（5 部分）
□ 方案评审
□ 逐项实现 TODO（Edit 工具），TDD 循环
□ 数据完整性设防（哨兵值 + 自检）
□ 走过分层测试闸门
□ 编译/语法验证
□ 功能验证（每步验收标准）
□ 集成验证
□ 三 C 判定（一致性 + 完整性 + 正确性）
□ Code Review
□ 同步到工作历史 + 知识持久化
```

## 检查清单

### 标准检查（每次改动必查）

```
□ 编译/语法检查通过（python -c import / lint）
□ 主路径功能正常（至少一个正向用例）
□ 至少一个边界条件已测试
□ 量化验证结果已记录
□ 与基线对比已明确（改了什么比什么）
□ 代码仅通过 Edit 工具修改
□ 敏感信息未泄露
□ 操作指引已编写
□ 数据完整性设防（哨兵值 + FAIL 级自检）
□ 分层测试闸门已通过
□ 三 C 验证已通过
□ 工作历史文档已更新
□ 关键发现已写入 memory 知识库
```

### 三 C 判定清单

```
一致性：
□ 实现与操作指引中的方案一致
□ 无"惊喜"逻辑（指引未讨论的实现）
□ 参数/配置与设计一致

完整性：
□ TODO list 全部实现
□ 边界情况已处理
□ 错误处理已覆盖
□ 配套产出齐全（注释/文档/配置/诊断）
□ 回退/关闭机制存在

正确性：
□ 编译/导入通过
□ 功能验证通过
□ 集成验证通过
□ 量化指标在预期范围内
```

### 诊断检查清单

```
□ 多维度证据已收集（数据输出/可视化/日志/配置/数据流/API文档）
□ 每个候选根因有明确假设和证据
□ 根因层级已分类（致命/严重/数据陈旧/呈现缺陷）
□ 修复优先级已确定
□ 未跳级归因（实现错误 → 方案偏差 → 方案不适用）
```

## 反模式清单

### 规划阶段

| 反模式 | 后果 | 正确做法 |
|--------|------|---------|
| 没有计划直接写代码 | 方向性错误、返工 | 先写操作指引，获得共识后再动手 |
| 计划太粗 | 遗漏关键步骤、无法对照检查 | 分解到 10-30 分钟可完成的粒度 |
| 计划不考虑回退 | 失败时无法恢复 | 每阶段前备份，冒烟先行 |
| 计划不写验收标准 | 不知道何时算完成 | 每个子步骤必须有可量化验证产出 |
| 跳过诊断直接修 | 治标不治本，问题复发 | 先确认根因再动手（阶段 1） |

### 实现阶段

| 反模式 | 后果 | 正确做法 |
|--------|------|---------|
| 先写实现后补测试 | 测试覆盖不完整、偏误 | TDD：先写测试确认失败，再写实现 |
| 修复症状不修根因 | 问题会以新形式复发 | 追到数据流/控制流的源头 |
| 同时改多个不相关的东西 | 无法隔离是哪个改动生效 | 一个变更单元只修一个根因 |
| 忽略哨兵值设计 | 0 和"未测量"无法区分 | 定义明确的哨兵值并全链路遵守 |
| 凭感觉验证 | "看起来好了"、"好像快了" | 保留基线，记录量化指标 |
| 一次改太多 | 一个 commit 改 >5 个不相关模块 | 一次 commit 只做一件事 |
| 无 profiling 盲目优化 | 优化了占比 <5% 的代码，净收益微乎其微 | 先用性能分析工具定位热点，再看决策树 |
| 优化后跳过功能等价验证 | "改了几行而已不用测" | L1→L2→L3 逐级验证 |
| 过早优化非热点 | 核心功能未完成就开始优化 | 先正确运行，再用 profiling 找瓶颈 |
| 优化后接口不兼容 | 改了函数签名或返回值类型 | 优化前后接口保持一致 |
| 过度复杂的优化 | 为 5% 的性能提升引入 200 行晦涩代码 | 权衡收益与复杂度，必要时放弃微优化 |
| 优化后移除原注释 | 优化完删掉原有注释和文档 | 保留注释，必要时增加优化说明 |
| 跳过小数据测试 | 只在大数据集上验证，忽略边界 | 每个优化至少覆盖 3 种数据规模 |

### 验证阶段

| 反模式 | 后果 | 正确做法 |
|--------|------|---------|
| 测试通过了就认为没问题 | 测试可能覆盖不足或阈值太宽 | 至少包含"防止回归全零"类哨兵测试 |
| 只看代码不看输出文件 | 代码逻辑正确但数据流有问题 | 验证输出文件中的实际值 |
| 跳过特定硬件测试 | 硬件相关的 bug 未被发现 | 无对应硬件环境用纯函数测试覆盖核心逻辑 |
| 验证完不汇总偏差报告 | 遗漏的步骤被遗忘 | 逐项对照计划生成偏差清单并分级 |
| 跳跃式排查 | 结果异常 → 直接怀疑方案不行 | 按归因优先级逐级排查（P1→P2→P3） |
| 跳过三 C | 改完"看起来跑通"就提交 | 三 C 是落地最低门槛 |
| 忽视边界 | 只测主路径 | 每个改动至少测 1-2 个边界条件 |

### 收尾阶段

| 反模式 | 后果 | 正确做法 |
|--------|------|---------|
| 只修代码不写文档 | 几个月后没人知道为什么这样写 | 更新项目概要和 API 使用指南 |
| 知识留在对话里 | 下次会话从零开始 | 写入 `memory` 持久化知识库 |
| 自动提交代码 | 操作者不知道改了什么 | 提交前呈交操作者审批 |
| 不记录基线对比 | 无法证明改动有效 | 至少记录 1-2 个关键指标的改前改后数据 |

## 附录 A：合成测试数据设计原则

1. **Ground truth 必须是解析解**：不依赖任何算法计算出的"真值"，而是由数学变换直接推导
2. **运动是平移的**：平移是唯一每个人/每个算法都能一致理解的运动类型
3. **输入有足够特征**：无特征区域使算法无定义——使用确定性噪声确保特征丰富
4. **确定性随机**：固定种子确保每次运行相同的"随机"输入
5. **子像素精度**：位移使用浮点数，用带插值的仿射变换生成帧

## 附录 B：GPU 计算开发专项注意事项

1. **异步流同步**：等待一个操作完成不等于等待后续操作完成——每个异步入队操作都需要独立的同步点
2. **缓冲区类型匹配**：不同算法输出不同数据类型。预分配错误类型会导致全零或错误值——直接验证输出 dtype
3. **计算方向约定**：不同 API 的输入/参考帧参数方向定义可能不同——需对照测试确认
4. **定点格式转换**：定点数格式的值需除以正确的缩放因子转换为浮点，偏移量范围需在测试中覆盖
5. **测试永不为零**：任何计算密集型测试必须包含"输出非全零"断言——它是静默故障的终极防呆

## 迭代机制

本 skill 需要在项目实践中持续优化。当以下情况出现时（无论哪个项目），会通过对应项目的特化版触发更新流程：

1. **发现新的有效工作方法** — 某类问题的排查顺序、验证策略被验证有效
2. **识别出新的反模式** — 反复出现的错误模式值得纳入文档
3. **发现准则空白** — 已有文档没有覆盖到的场景或环节

更新流程：
1. 先写入对应项目的特化版（含项目特化表述）
2. 再去除特化表述，同步到本通用版

每次更新更新版本号和变更说明。
